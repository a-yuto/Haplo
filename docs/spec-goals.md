# Haplo 言語仕様 — 目標・要件・アーキテクチャ

## §0 言語哲学 / 設計原則

迷ったときの優先順位を決める道具。各原則には「それが何を決めるか」を添える。

1. **実行時例外を可能な限り無くす** — エラーは実行時ではなく型検査で先に捕まえる。shape 不一致もコンパイル時に弾く。依存型はこの原則の手段。
2. **コンパイラは味方** — エラーメッセージは人に優しく、原因と直し方を示す。型検査は「邪魔」ではなく「ガイド」。
3. **一つのことは一つのやり方でしか出来ないようにする（なるべく）** — 同じ操作に複数の構文・記法を用意しない。例：転置は `transpose` 一本に絞り `.T` は作らない。
4. **シンプルな機能を組み合わせて使う** — 大きな専用機能より、小さく直交したプリミティブの合成を優先。
5. **冗長なのは悪いことではない** — 明示性を簡潔さより優先する。型注釈や明示的な記述を厭わない。

> 系譜：1・2 は Elm、3 は Python の Zen、4 は Unix 的な合成、5 は Go 的な明示性。

## §2 機能要件

### 2.1 テンソル・数値計算

スカラー/ベクトル/行列/高階テンソルの統一表現。要素ごと演算とブロードキャスト、行列積・転置・reshape・スライス・集約（sum, mean, max …）、数学関数（exp, log, tanh, softmax 等）。計算の実体は ndarray に委譲する。

### 2.2 自動微分（中核）

リバースモード（誤差逆伝播）。演算を記録するテープ（計算グラフ）を自作し、逆向きに辿って勾配を算出する。`grad f` は「関数を受け取り勾配関数を返す」高階関数として提供。MVP では 1 階微分で可。

### 2.3 型システム（最重要・最難関）

テンソルの shape を型レベルに持ち上げ、次元不一致をコンパイル時に弾く。次元変数による shape 多態、dtype の静的チェック、shape 推論、shape 算術（`m+n` 等）、値依存の型（dependent function type）を備える。

---

## §4 非機能要件

- **性能**：配列演算は ndarray 任せで実用速度を確保。グラフ最適化・演算融合は将来課題
- **正しさ**：型・shape 検査の健全性を最優先（バグを実行前に弾くことが本 DSL の価値）
- **エラーの分かりやすさ**：shape 不一致時に「期待 vs 実際」の次元を明示
- **移植性**：純 Rust 構成を優先

## §5 アーキテクチャ概観

```
ソース
  │ lexer（改行=区切り、列追跡なし）
  ▼ トークン列
  │ parser → AST
  ▼ AST
  │ 型検査＋shape検査（依存型チェッカ）★最難関
  ▼ 型付きAST
  │ 評価（インタプリタ）
  ├─ ndarray を呼んで計算
  └─ テープ記録 → 逆伝播（reverse-mode autodiff）
```

主要部品：(1) lexer/parser、(2) 依存型チェッカ（次元の単一化・shape 算術・shape 推論）、
(3) 評価器、(4) autodiff テープ、(5) ndarray ラッパ。
